Catalog of Technologies and Line Proposals - Documentation

Introduction
The Catalog of Technologies and Line Proposals is a Flask-based web application that connects to a SQL Server database to present technology capability tables and proposal tables in a user-friendly interface. It offers pages to browse technologies, search by capabilities (PCB sizes, thickness, weight), view proposals with rich tooltips and images, and perform admin tasks such as editing tables and exporting to Excel.

What is the Catalog App
Purpose: Provide a centralized, easy-to-use portal for viewing technology data, exploring proposals, and performing controlled admin operations on the underlying SQL tables.
Scope: Renders capability tables and proposal tables, supports filtering and search, assembles informative tooltips and images for equipment columns, and exposes admin endpoints for reading, exporting, updating, creating, renaming, and deleting tables.

Context
This app standardizes how users access and review technology information across multiple tables in a single UI. It reduces reliance on ad-hoc queries and manual spreadsheet prep by offering a consistent rendering, inline documentation links, and built-in export options. It also introduces app-level admin users distinct from DB credentials for safer operational controls.

Background
Previously, users consulted multiple isolated tables and documents. The app aggregates technology tables, derives friendly names for groups (e.g., Screen Printer, Pick & Place), and shows proposals horizontally with optional images. It adds search filters for PCB sizes, thickness, and weight, and bridges proposal content to technology capabilities via heuristic matching.

How it works
Home (index):
Loads mapping from [Tecnologias_Tecnologias] and derives a unique set of friendly technology names using simple heuristics.
Shows a dropdown for Tecnología; selecting renders content via /tecnologia or inline via /tecnologia_inline.

Tecnología page (/tecnologia, /tecnologia_inline):
Iterates a curated list of TECH_TABLES (capability and proposal tables).
Supports "ALL" and grouped selection: Documento='GROUP:<friendly>' aggregates multiple tables under the same friendly technology.
Converts each table to HTML via pandas, groups results by friendly name, and returns combined sections.

Proposals page (/propuestas):
Discovers all base tables whose name contains 'Proposal' and builds a proposal card per table.
Drops ID-like columns, and transforms two-column vertical tables (Equipo, Costo) into a single-row horizontal layout.
Inserts an image row under headers: prefers explicit image-like values in the table, otherwise maps headers to technology/vendor images.
Builds rich tooltips per equipment header: Documento(s), Min/Max PCB sizes (including M/L/XL variants for conveyors), Thickness, Weight, and extra fields for Camalot.
Special handling for provenance columns (Usuario, Categoria, Fecha de creacion) showing user and date icons.

Search page (/search):
Accepts capability filters: PCB size ranges (X/Y), optional single-value legacy fields, thickness min/max, weight min/max.
Optionally filters by technology and document text.
Searches only capability tables (excludes proposals). Applies structured size checks using min/max or generic size-like columns.
After capability matches, seeks related proposals and adds those whose equipment headers match the same filters via a helper that scans tech tables.
Renders results grouped by friendly technology name and shows proposal cards inline.

Documentation page (/documentacion):
Renders [Tecnologias_Tecnologias] with styled rows. If a URL-like column is present, displays a link button for convenient navigation.

Admin functions:
/admin (requires app-level admin session) shows the admin UI.
/admin/login validates against internal APP_ADMINS (not DB credentials).
/admin/tables lists base table names; /admin/table fetches a table preview with schema-aware resolution.
/admin/export_table_excel: export a single table to Excel; /admin/export_all_tables_excel: export all tables grouped by Tecnología into one workbook.
/admin/update updates a cell using a WHERE clause built from an original row snapshot.
/admin/create_table creates a new NVARCHAR(MAX)-based table; optional Tecnología prefixing.
/admin/alter_table supports adding/removing columns, renaming tables, inserting rows, deleting specific rows, and updating Tecnología mappings.
/admin/delete_table drops a table after validation.
/admin/technologies lists distinct Tecnología values; /admin/tech_map returns mapping Tecnología -> [Documento].
/admin/tables_for_technology finds tables associated with a selected Tecnología using multiple heuristics (direct Documento mapping, prefix matching, friendly-name mapping, Documento column checks).
/admin/users lists usernames from Usuarios.

Images and tooltips:
resolve_tech_image_src(header): maps equipment header text to static images (e.g., KohYoung AOI, SPI, Camalot, NXT, Conveyor, Reflow, Screen Printer, Pick & Place).
detect_image_in_column(df, col): if a column contains an explicit image path or data URI, uses it.
Tooltips pull sizes/thickness/weight from tech tables via get_equipment_pcb_sizes (with caching), plus document links and Camalot extras.

Code
Main files:
app.py: Flask app with routes, capability/proposal rendering, search filters, admin endpoints, image/tooltip logic, and size extraction helpers.
db.py: pyodbc connection setup, DB credentials, application-name injection, admin credential validation, and app-level admin users.
run_app.py: single-process server runner that opens the browser and runs without the reloader (friendly for PyInstaller builds).
Supporting assets:
templates/: HTML templates (index.html, tecnologia.html, propuestas.html, search.html, documentacion.html, admin.html).
static/: CSS/JS/images. JS includes admin interactions and UI helpers.
Catalogo_Debug.spec: PyInstaller spec for packaging.

Entry point
Development: app.py includes __main__ to open the browser and run the server (debug=True, no reloader).
Single-run: run_app.py starts the app with debug=False and no reloader, suitable for packaging.

Configuration
Database connection in db.py: SERVER, DATABASE, UID, PWD, DRIVER ('ODBC Driver 17 for SQL Server').
Optional APP_NAME included in the connection string for easier tracing; configurable via environment.
Timezone for proposal creation timestamps via APP_TZ (defaults to America/Mexico_City). Uses ZoneInfo when available.

Usage Notes
Dependencies: Flask, pandas, pyodbc, xlsxwriter (Excel), openpyxl (fallback), werkzeug.security.
ODBC Driver 17 for SQL Server must be installed and the DB reachable from the host.
Admin auth is application-level (APP_ADMINS), separate from SQL credentials; keep app.secret_key secure in production.
Proposals created via /create_proposal automatically include provenance columns 'Usuario' and 'Fecha de creacion' (DATETIME2, local time).
Large tables are previewed with row limits in admin endpoints; exports write sheets by Tecnología and include table headers.

Architecture and Components
Backend (Flask): Routing, Jinja2 rendering, search and aggregation logic, admin endpoints, Excel exports.
Data (SQL Server): Technology capability tables and proposal tables; mapping table [Tecnologias_Tecnologias]; users table [Usuarios] (for end-user login, separate from the app admin).
Connection (pyodbc): Connection string defined in db.py with DRIVER, SERVER, DATABASE, UID, PWD; optional APP_NAME for SQL Server tracing.
Frontend (templates + static): Templates in templates/, styles in static/style.css, scripts in static/js/ (admin.js, loading.js, popover.js, proposal_builder.js), images in static/img/.
In-memory cache: PCB size results per equipment (`_EQUIP_SIZE_CACHE`) with configurable TTL (SIZES_CACHE_TTL, default 1800s).

Routes and Contracts (API/UI)
GET /: Home page. Loads friendly technology names (derived from [Tecnologias_Tecnologias]) and shows a selector.
GET /manual: Simple user manual page.
POST /tecnologia: Renders technology tables based on selected technology/document.
	- Parameters (form): tecnologia, documento (can be "ALL" or "GROUP:<friendly>").
	- Output (HTML): Sections grouped by friendly name with rendered tables.
POST /tecnologia_inline: Same as /tecnologia but returns JSON for inline rendering on the index.
	- Input: form or JSON {tecnologia, documento}.
	- Output: { tecnologia, documento, resultados: { friendly: html_table } }.
GET/POST /search: Capability search (PCB X/Y ranges, thickness, weight) with optional technology/document filters.
	- Input (form): pcb_min_x, pcb_min_y, pcb_max_x, pcb_max_y; pcb_min_size/pcb_max_size (legacy); thickness_min/thickness_max; weight_min/weight_max; tecnologia; documento.
    	- Output (HTML): Results by technology plus related proposals with rich tooltips.
GET /propuestas: Discovers Proposal_* tables and renders them as horizontal cards with an image row and tooltips.
GET /documentacion: Renders [Tecnologias_Tecnologias] with per-row link buttons detected from URL-like columns.
POST /image_for_headers: Returns a mapping header->img src for column headers, preferring explicit images found in Proposal tables; otherwise falls back to heuristics.
POST /admin/login: App admin login (APP_ADMINS). JSON response {success}.
GET /admin: Admin UI (requires admin session).
GET /admin/tables: Lists base tables. JSON {tables}.
GET /admin/table?name=...: Table preview (TOP 500) with schema resolution. JSON {columns, rows, tecnologia}.
GET /admin/export_table_excel?table=...: Exports a single table to Excel (in-memory download).
GET /admin/export_all_tables_excel: Exports all tables grouped by Tecnología into a single .xlsx.
POST /admin/update: Updates a cell value using a WHERE built from the original row snapshot. JSON {success, rows_affected}.
POST /admin/create_table: Creates a table with NVARCHAR(MAX) columns. Optional prefix by Tecnología. JSON {success}.
POST /admin/alter_table: Renames table (sp_rename), adds/drops columns, inserts rows, deletes rows, updates mappings. JSON {success}.
POST /admin/delete_table: Drops a table after validation. JSON {success}.
GET /admin/technologies: Lists distinct Tecnología values. JSON {technologies}.
GET /admin/tech_map: Mapping Tecnología -> [Documento]. JSON {map}.
GET /admin/tables_for_technology?tech=...: Heuristic to identify tables associated with the selected technology. JSON {tables}.
GET /admin/users: Lists users from [Usuarios]. JSON {users}.
POST /user/create_account: Creates a user in [Usuarios] with password hash; attempts to add PasswordHash column if missing. JSON {success}.
POST /user/login: Authenticates a user from [Usuarios] with hash verification. JSON {success, user}.
POST /create_proposal: Creates Proposal_<name> with selected content columns and provenance columns (Usuario, Fecha de creacion). JSON {success, table}.

Data Sources and Normalization
TECH_TABLES: Curated list of technology and proposal tables; meta tables like Tecnologias_Tecnologias and Usuarios are excluded from capability searches.
[Tecnologias_Tecnologias]: Maps Documento -> Tecnología (used for friendly names, grouping, and filters).
[Usuarios]/[dbo].[Usuarios]: Stores end-user credentials (not to be confused with APP_ADMINS).
Friendly name normalization: `get_friendly_name()` applies heuristics (Pick & Place, Reflow Oven, Screen Printer, SPI, AOI & AXI, Camalot, Conveyor, NXT, etc.).
Comparison normalization: `_normalize_key()` reduces strings to lowercase alphanumeric without separators.

Measurement Extraction and Tooltips
Size algorithm: `get_equipment_pcb_sizes()` scans technology tables (excludes proposals), infers PCB min/max (and M/L/XL variants for conveyors), thickness, and weight from candidate columns (PCB Min/Max Size, Part Size, Work/Dispense/Substrate/Board Area, thickness*, weight*/kg).
Matching heuristics: Contains and token-normalized matching to locate relevant rows; avoids false positives; limits TOP n for performance.
Cache TTL: `get_equipment_pcb_sizes_cached()` stores results per header/equipment to speed up proposals and search rendering.
Images per header: `detect_image_in_column()` prioritizes explicit content (data URI, /static/, http/https, common image extensions). `resolve_tech_image_src()` maps images by keywords when no explicit image is present.
Rich tooltips: Documents, PCB Min/Max (generic and variants), thickness (min/max/value), weight (max/value), Camalot extras (conveyor type, clearance, transport height, etc.), plus intrinsic table columns when key info is missing.

Security and Sessions
App admin: APP_ADMINS defined in db.py; authentication separate from SQL credentials.
End users: Accounts managed in [Usuarios] with password hashing (werkzeug.security), verified in /user/login.
Flask session: `app.secret_key` (set a secure value in production). Session flags `admin_authenticated`, `user_authenticated`, `user`.
SQL identifier validation: `_valid_identifier()` and `_sanitize_identifier()` to avoid injections in table/column names.
Dangerous operations: Admin endpoints perform extra validations (table existence, columns, sp_rename for renames, DROP checks).

Configuration and Environment Variables
db.py: SERVER, DATABASE, UID, PWD, DRIVER ("ODBC Driver 17 for SQL Server").
APP_NAME: Application name for tracing (Application Name and APP in the connection string).
APP_TZ: Time zone for proposal timestamps (default America/Mexico_City).
SIZES_CACHE_TTL: TTL for size cache (seconds, default 1800).

Local Execution
Requirements: Install ODBC Driver 17, access to SQL Server, Python 3.9+ recommended (ZoneInfo available), packages Flask/pandas/pyodbc/xlsxwriter/openpyxl.
Typical commands:
	- Development (opens browser automatically):
		python app.py
	- Run without reloader (good for packaging):
		python run_app.py
Ports: Defaults to 5000 (host 127.0.0.1). Change if occupied.

Packaging and Deployment
PyInstaller: Spec file (Catalogo_Debug.spec) to build a single-file executable.
Recommendations: Disable reloader, use debug=False, include static assets in the package.
Service/host: For production, consider a WSGI server (gunicorn/waitress) behind a reverse proxy; adjust ODBC connection/credentials and `app.secret_key`.

Performance and Limits
Table reads: Row counts are limited (TOP) in previews and searches for performance.
pandas->HTML rendering: For very large tables, consider pagination or summaries.
Excel export: Groups by Tecnología and inserts a table header; use xlsxwriter (openpyxl fallback).

Troubleshooting
ODBC/driver not found: Install "ODBC Driver 17 for SQL Server" and verify the driver name matches db.py.
Connection refused/timeout: Validate SERVER/DATABASE/UID/PWD; test connectivity from the host; check firewalls.
run_app.py exits with code 1: Check console logs; common causes are missing Flask/pyodbc/pandas or connection errors; install dependencies and confirm connection string (avoid leaking PWD in logs).
Port 5000 in use: Change `port` in app.run or stop the occupying process.
Images not showing: Ensure /static/img/ paths and filenames exist; review `resolve_tech_image_src()` and `detect_image_in_column()`.
Empty tooltips: Verify measurement columns and table mappings; adjust TECH_TABLES and heuristics if columns changed.

Limitations and Considerations
Heuristics: Column detection (size/thickness/weight) relies on keywords; column renames may require adjustments.
Incomplete data: When sizes are missing, the app falls back to intrinsic proposal info; may be partial.
Security: Admin endpoints should be used by authorized staff only; SQL credentials are not exposed.

Glossary
Technology: Functional group (e.g., Screen Printer, Pick & Place, Reflow Oven).
